
# Mastering Git Branching and Conflict Resolution: A Hands-On Lab



Welcome to this practical guide on **Git Branching and Conflict Resolution**. This repository is designed as a self-paced learning module for developers to understand how isolated development streams work, how to manage Pull Requests, and how to confidently resolve the dreaded merge conflict.

## 🎯 Learning Objectives

By the end of this lab, you will understand:

* How to create and navigate between isolated branches.
* The mechanics of a Pull Request (PR) and merging.
* How to manually resolve merge conflicts when automatic merging fails.

---

## 🧠 Core Concepts at a Glance

| Concept | What it actually means | Why we use it |
| --- | --- | --- |
| **Branch** | An independent timeline of commits. | To build experimental features without breaking the main production code (`master`/`main`). |
| **Checkout** | Switching your workspace to a different branch. | To safely move between features or tasks without mixing code. |
| **Pull Request (PR)** | A request to merge your branch into another. | Allows team members to review, discuss, and approve changes before deployment. |
| **Merge Conflict** | A state where two branches edit the exact same line of code. | Triggers a safety block so developers can manually decide which code is correct. |

## 🛠️ Step-by-Step Lab Exercise

### Scenario: Managing the Team Content Plan

Imagine you are managing a repository for a development team's educational content. The file `rencana_konten.txt` lists the upcoming topics. You need to add new frameworks using isolated feature branches.

---

### Step 1: Create a Feature Branch (`Feature-A`)

Instead of editing the main production track, create an isolated workspace.

1. In your repository, click the branch dropdown (usually labeled `master` or `main`).
2. Type a new branch name: `feature-javascript-frameworks`.
3. Click **Create branch**.

> **Note:** You have just performed a **Checkout**. Any changes you make now will be safely locked inside this specific branch.

#### Task 1:

In this branch, modify `rencana_konten.txt` to add modern JavaScript frameworks:

```text
- JavaScript Fundamentals
- Framework Vue.js
- Framework React

```

Commit this file directly to your feature branch with the message: `"Add Vue and React to JS track"`.

---

### Step 2: Simulate Parallel Teamwork (`Feature-B`)

To simulate a real-world conflict, we need a second developer (or branch) changing the *same* section of the file before `Feature-A` is finished.

1. Switch your repository dropdown back to the `master` branch.
2. Create another branch named `feature-alternative-frameworks`.
3. Edit the *exact same line* in `rencana_konten.txt` but with different choices:

```text
- JavaScript Fundamentals
- Framework Angular
- Framework Svelte

```

4. Commit this with the message: `"Add Angular and Svelte options"`.

---

### Step 3: The Smooth Merge (Fast-Forward)

Now let's bring the first feature branch back into the main project.

1. Switch back to your `master` branch.
2. Click **Compare & pull request** next to `feature-javascript-frameworks`.
3. GitHub will analyze the file and state: `✔️ Able to merge`.
4. Click **Create pull request**, review the changes, and click **Merge pull request**.

Your `master` branch now contains the Vue and React updates.

---

### Step 4: Facing the Merge Conflict

Now, let’s try to bring the second feature branch (`feature-alternative-frameworks`) into `master`.

1. Go to the **Pull Requests** tab.
2. Open a new PR from `feature-alternative-frameworks` into `master`.
3. **The Result:** GitHub blocks the merge and warns:
> ❌ *This branch has conflicts that must be resolved.*



#### Why did this happen?

Because both branches modified the lines directly underneath `JavaScript Fundamentals`. Git doesn't know whether you want to keep Vue/React, Angular/Svelte, or both.

---

### Step 5: Resolving the Conflict Manually

1. Click the **Resolve conflicts** button in the Pull Request UI.
2. Look at the conflict markers generated by Git:

```text
<<<<<<< master
- Framework Vue.js
- Framework React
=======
- Framework Angular
- Framework Svelte
>>>>>>> feature-alternative-frameworks

```

3. **Your Task:** Manually edit the file inside the editor to clean it up. Decide what code stays. Let's say you want to keep all four options. Delete the markers (`<<<<<<<`, `=======`, `>>>>>>>`) and organize the text cleanly:

```text
- Framework Vue.js
- Framework React
- Framework Angular
- Framework Svelte

```

4. Click **Mark as resolved**, commit the merge, and finalize the Pull Request.

---

## 🏁 Summary of Best Practices

* **Keep Branches Focused:** One branch should do exactly one thing (e.g., fix a single bug or add a specific feature).
* **Pull Frequently:** Always update your local `master` branch before creating a new feature branch to minimize conflicts later.
* **Communicate:** If a merge conflict happens, talk to the teammate who wrote the conflicting line to ensure you don't delete essential logic.
